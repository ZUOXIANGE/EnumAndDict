## 字典与枚举在API开发中的统一处理方案 - ASP.NET Core 10 设计文档

### 一、核心概念与场景区分

| 概念             | 定义                     | 特征                               | 典型场景           |
| ---------------- | ------------------------ | ---------------------------------- | ------------------ |
| **枚举（Enum）** | 编译期固定的命名常量集合 | 值不可变、数量少、类型安全         | 订单状态、错误码   |
| **字典（Dict）** | 可动态维护的键值对集合   | 值可变更、数量可多可少、需后台管理 | 性别、国家、配置项 |

**核心原则**：
- 业务逻辑强依赖、不会变化的值 → 枚举
- 可能被用户/管理员动态维护的值 → 字典表

---

### 二、枚举的前端下拉框数据获取方案

#### 2.1 设计目标
前端不能硬编码枚举值，必须通过后端接口动态获取。

#### 2.2 推荐方案：独立枚举查询接口

**接口设计规范**：
```
GET /api/enums/{enumName}
```

**响应格式**：
```json
{
  "code": 200,
  "data": [
    { "value": 0, "label": "待支付", "name": "PENDING" },
    { "value": 1, "label": "已支付", "name": "PAID" }
  ]
}
```

**字段说明**：
- `value`：实际存储到数据库的值（整数或字符串）
- `label`：前端展示的文本
- `name`：枚举名称

#### 2.3 .NET 实现要点

**技术选型**：
- 使用 `System.Reflection` 读取枚举类型信息
- 配合 `DisplayAttribute` 或 `DescriptionAttribute` 存储中文标签
- 编写枚举扩展方法获取标签文本

**核心类型**：
- `EnumInfoService`：负责读取所有枚举及成员信息，使用 `IMemoryCache` 缓存
- `EnumOptionResponse`：枚举选项的响应 DTO
- `EnumController`：提供 API 端点

**缓存策略**：
- 应用启动时使用 `IHostedService` 预加载所有枚举信息(可选)
- 枚举值不会变，永久缓存

#### 2.4 备选方案

| 方案             | 适用场景            | 说明                           |
| ---------------- | ------------------- | ------------------------------ |
| 统一枚举接口     | 枚举数量少（<10个） | 一次返回所有枚举，减少请求次数 |
| 业务接口附带枚举 | 特定表单页面        | 查询数据时同时返回可用选项     |

使用 https://github.com/xin9le/FastEnum 库避免使用反射获取相关的信息,提高性能.

### 三、字典翻译方案（唯一方案：响应拦截器）

#### 3.1 需求描述
在 DTO 或 ViewModel 中，通过**特性（Attribute）**自动完成字典值的翻译。

**使用示例**：
```csharp
public class UserResponse
{
    [DictTranslate(DictCode = "sex", TargetProperty = nameof(SexDesc))]
    public int Sex { get; set; }           // 原始值：0或1
    
    public string SexDesc { get; set; }    // 自动填充翻译结果
}
```

#### 3.2 核心组件

| 组件                     | 职责                                                  |
| ------------------------ | ----------------------------------------------------- |
| `DictTranslateAttribute` | 标记需要翻译的字段，指定字典类型和目标属性名          |
| `DictService`            | 字典数据查询服务，含缓存机制                          |
| `DictTranslateFilter`    | 实现 `IAsyncResultFilter`，在响应返回前拦截并执行翻译 |
| `ObjectVisitor`          | 递归遍历对象树，收集待翻译项并执行赋值                |
| `DictTranslationHelper`  | 执行批量字典查询和属性赋值                            |

#### 3.3 特性属性设计

| 属性             | 类型   | 必填 | 说明                                   |
| ---------------- | ------ | ---- | -------------------------------------- |
| `DictCode`       | string | 是   | 字典类型编码，如 "sex"、"nation"       |
| `TargetProperty` | string | 是   | 翻译结果存放的目标属性名，如 "SexDesc" |
| `DefaultValue`   | string | 否   | 翻译失败时的默认值，默认为原始值       |

#### 3.4 架构设计

```
请求 → Controller → Action 执行 → ObjectResult → DictTranslateFilter → 递归翻译 → 返回响应
                                    ↑
                                    |
                            IAsyncResultFilter
```

**核心流程**：
1. 过滤器中获取 Action 返回的 `ObjectResult` 的 `Value`
2. 调用 `ObjectVisitor` 递归遍历对象树
   - 识别集合类型 → 递归处理每个元素
   - 识别复杂类型 → 反射获取所有属性
   - 检查属性是否带 `DictTranslateAttribute`
3. 收集所有需要翻译的 `(DictCode, 原始值)` 到集合
4. 批量调用 `DictService` 查询翻译结果
5. 通过反射将翻译结果赋值给目标属性
6. 返回修改后的对象

#### 3.5 功能要求

**基础功能**：
- [ ] 支持单层对象的字典翻译
- [ ] 支持 `List<T>`、`IEnumerable<T>`、数组等集合类型中元素的翻译
- [ ] 支持嵌套对象的递归翻译（如 `OrderResponse` 中包含 `UserResponse`）
- [ ] 支持字典嵌套（如对象 A 的属性是对象 B，B 中也有带字典注解的属性）
- [ ] 目标属性不存在时的异常处理（降级/日志）

**缓存设计**：
- [ ] 使用 `IMemoryCache` 存储字典映射
- [ ] 支持缓存过期/主动刷新机制
- [ ] 可选：`IDistributedCache`（Redis）作为二级缓存
- [ ] 提供缓存刷新的 API 端点

**字典数据源**：
- [ ] 从数据库字典表加载（EF Core 查询）
- [ ] 支持应用启动时预加载（`IHostedService`）
- [ ] 支持运行时字典变更后的缓存刷新

**高级特性**：
- [ ] 支持自定义翻译失败时的默认值（通过 `DefaultValue` 属性）
- [ ] 支持根据上下文条件翻译（如根据请求中的语言偏好返回不同文本）
- [ ] 支持链式翻译（字典字段的翻译结果再作为另一个字典的输入）

**性能要求**：
- [ ] 避免循环引用导致的无限递归（使用 `HashSet<object>` 配合 `ReferenceEqualityComparer`）
- [ ] 限制递归深度（可配置，如默认 10 层）
- [ ] 批量翻译优化：收集所有需要翻译的值，一次性查询数据库，避免 N+1 问题

#### 3.6 异常处理策略

| 异常场景         | 处理策略                                      |
| ---------------- | --------------------------------------------- |
| 目标属性不存在   | 记录警告日志，跳过本次翻译，不影响主流程      |
| 字典编码不存在   | 使用 `DefaultValue`（若有）或原始值，记录警告 |
| 反射访问失败     | 记录错误日志，降级处理                        |
| 递归深度超限     | 停止继续递归，记录错误                        |
| 翻译值类型不匹配 | 尝试类型转换，失败则使用默认值                |

### 四、响应拦截器实现要点

#### 4.1 类型判断规则

| 类型                                       | 处理方式                   |
| ------------------------------------------ | -------------------------- |
| 简单类型（`int`、`string`、`DateTime` 等） | 跳过，不递归               |
| 集合类型（`IEnumerable`、`IList`、数组等） | 遍历每个元素，递归处理     |
| 字典类型（`IDictionary`）                  | 遍历每个 `Value`，递归处理 |
| 复杂类型（普通 class/record）              | 反射获取属性，检查字典注解 |

#### 4.2 批量翻译优化流程

```
1. 遍历对象树，收集所有翻译请求
   → List<TranslationRequest> 
   → 每个请求包含: { 对象引用, 原始值, DictCode, 目标属性名, 默认值 }

2. 按 DictCode 分组，批量查询数据库
   → Map<DictCode, Map<原始值, 翻译结果>>

3. 遍历翻译请求列表，反射赋值
```

#### 4.3 循环引用检测

使用 `HashSet<object>`（配合 `ReferenceEqualityComparer`）记录已处理的对象引用，遇到重复引用时跳过，避免无限递归。

### 五、依赖注入配置设计

#### 5.1 服务注册

```csharp
// Program.cs 或服务扩展方法

// 基础服务
services.AddMemoryCache();                                    // 字典缓存
services.AddScoped<IDictService, DictService>();              // 字典查询服务
services.AddScoped<DictTranslationHelper>();                  // 翻译辅助类
services.AddScoped<DictTranslateFilter>();                    // 响应过滤器

// 枚举服务
services.AddSingleton<IEnumInfoService, EnumInfoService>();   // 枚举信息服务

// 全局注册过滤器（推荐）
services.AddControllers(options =>
{
    options.Filters.Add<DictTranslateFilter>();
});

// 或者使用 [ServiceFilter] 特性单独控制（不推荐，需要每个 Controller 或 Action 手动添加）
```

#### 5.2 配置选项

```csharp
public class DictTranslateOptions
{
    public int MaxRecursionDepth { get; set; } = 10;                    // 最大递归深度
    public bool EnableBatchTranslation { get; set; } = true;            // 是否启用批量翻译
    public int CacheDurationMinutes { get; set; } = 60;                 // 缓存时间（分钟）
    public bool ThrowOnMissingTargetProperty { get; set; } = false;     // 缺失目标属性是否抛异常
    public bool EnableCycleDetection { get; set; } = true;              // 是否启用循环引用检测
}

// 注册配置
services.Configure<DictTranslateOptions>(configuration.GetSection("DictTranslate"));
```

### 六、枚举与字典的混合使用场景

#### 6.1 数据库存储
- 枚举字段：存 `int` 或 `string` 值（推荐 `int`）
- 字典字段：存字典项的 `DictValue`

#### 6.2 API 响应格式

统一返回格式（推荐）：
```json
{
  "status": "Paid",
  "statusValue": 1,
  "statusLabel": "已支付",
  "gender": "1",
  "genderLabel": "男"
}
```

**命名规范**：
- 枚举原始值字段：使用属性名本身（如 `Status`）
- 枚举翻译字段：`{属性名}Label`（如 `StatusLabel`）
- 字典原始值字段：使用属性名本身（如 `Gender`）
- 字典翻译字段：通过 `TargetProperty` 自定义（如 `GenderDesc`）

#### 6.3 API 请求校验

**枚举字段**：
- 前端可提交 `string`（枚举名称）或 `int`（枚举值）
- 使用 `Enum.TryParse` 转换（忽略大小写）
- 转换失败返回 400 错误

**字典字段**：
- 前端提交字典项的 `DictValue`
- 使用 `DictService.Exists()` 确认值是否存在
- 不存在返回 400 错误

### 七、参考表结构

#### 7.1 字典表（SQL Server）

```sql
CREATE TABLE SysDict (
    Id BIGINT IDENTITY(1,1) PRIMARY KEY,
    DictCode NVARCHAR(32) NOT NULL,          -- 字典类型编码，如 'sex'
    DictValue NVARCHAR(32) NOT NULL,         -- 字典项原始值，如 '1'
    DictLabel NVARCHAR(64) NOT NULL,         -- 字典项显示文本，如 '男'
    SortOrder INT DEFAULT 0,                 -- 排序
    IsEnabled BIT DEFAULT 1,                 -- 状态：0禁用 1启用
    CreatedAt DATETIME2 DEFAULT GETUTCDATE(),
    UpdatedAt DATETIME2 NULL,
    
    CONSTRAINT UK_SysDict_DictCode_Value UNIQUE (DictCode, DictValue)
);

CREATE INDEX IX_SysDict_DictCode ON SysDict(DictCode);
CREATE INDEX IX_SysDict_IsEnabled ON SysDict(IsEnabled);
```

#### 7.2 字典缓存 Key 设计

```
dict:{dictCode}:{dictValue} → dictLabel
```

### 八、后续开发建议

#### 8.1 实现优先级

| 阶段        | 内容                                          | 预估复杂度 |
| ----------- | --------------------------------------------- | ---------- |
| **Phase 1** | 基础枚举查询接口 + 枚举扩展方法 + 单元测试    | 低         |
| **Phase 2** | 字典查询服务 + `IMemoryCache` + 从数据库加载  | 中         |
| **Phase 3** | `ObjectVisitor` 递归访问器 + 反射赋值基础功能 | 中         |
| **Phase 4** | `DictTranslateFilter` 响应拦截器集成          | 中         |
| **Phase 5** | 批量翻译优化 + 循环引用检测                   | 中高       |
| **Phase 6** | 异常处理、日志、配置选项                      | 低         |
| **Phase 7** | 性能测试、压力测试                            | 中         |
| **Phase 8** | 可选：Redis 二级缓存、缓存刷新 API            | 中         |

#### 8.2 测试用例清单

**功能测试**：
- [ ] 普通 DTO 翻译（单个字典属性）
- [ ] DTO 中有多个字典属性
- [ ] `List<T>` 翻译
- [ ] 嵌套对象翻译（2 层、3 层）
- [ ] 集合中包含嵌套对象
- [ ] 字典编码不存在的降级处理
- [ ] 目标属性不存在的降级处理
- [ ] 翻译值类型不匹配的处理
- [ ] 值为 null 的处理

**性能测试**：
- [ ] 单个对象翻译（1-10 个字典字段）→ 目标 < 1ms（缓存命中）
- [ ] 100 个对象的列表翻译 → 目标 < 10ms（批量查询）
- [ ] 缓存未命中时的首次查询 → 目标 < 50ms
- [ ] 1000 个对象的列表翻译 → 目标 < 100ms

**并发测试**：
- [ ] 100 并发请求，缓存未命中场景
- [ ] 1000 并发请求，缓存命中场景

#### 8.3 注意事项

1. **性能热点**：反射操作在首次调用后会进入缓存路径（`PropertyInfo` 可缓存），性能可接受
2. **批量查询**：务必实现批量翻译，避免循环内单次查询
3. **循环引用**：务必检测，否则复杂对象图可能导致栈溢出
4. **缓存失效**：字典数据变更时，需要提供主动清除缓存的机制
5. **日志记录**：翻译失败时记录日志，但不要影响主流程
6. **异步支持**：字典查询使用异步方法（`ToDictionaryAsync`），避免阻塞

#### 8.4 可扩展点

- 支持自定义字典数据提供者（接口 `IDictProvider`）
- 支持从配置文件、Redis、远程 API 加载字典
- 支持多语言翻译（根据 `Accept-Language` 请求头）
- 支持字典值格式化（如日期格式、数字格式）
